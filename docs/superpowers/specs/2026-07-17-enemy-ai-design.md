# 阶段 4 敌人 AI 设计

**日期：** 2026-07-17
**状态：** 已确认
**上游依赖：** 阶段 3 自动化、生产构建和九项真机出口全部通过
**范围：** 分层 AI 框架、三类灰盒敌人、三种可切换敌人遭遇、区域约束与确定性验收

## 1. 目标与边界

阶段 4 在阶段 3 固定 tick 战斗域上交付可扩展、可维护、可确定性测试的敌人 AI 基础。当前
实现必须支持三类敌人，并为阶段 5 首领、后续新敌人、召唤物和更复杂决策策略保留稳定接口。

本阶段交付：

- 感知、决策、战术规划、动作执行四层 AI 管线。
- 裂爪兽、辉印祭司、蚀甲守卫三类敌人。
- 单体裂爪兽、裂爪兽与祭司混合战、单体守卫三种可独立启动和复位的敌人遭遇。
- 训练遭遇适配器，保留阶段 3 的训练假人和训练脉冲验收环境。
- 战斗区域约束、敌人分离、不可达脱困、返回区域、打断、破韧和死亡生命周期。
- 多目标软锁定、灰盒渲染和诊断快照。

明确不属于阶段 4：

- NavMesh、群体编队、复杂动态避障、行为树编辑器或 GOAP。
- 掉落、战斗门、波次推进、失败重试和关卡流程，这些属于阶段 5。
- 正式敌人模型、动画、VFX、音频和最终 HUD，这些属于阶段 6。
- 网络同步、Mod 加载、脚本运行时或在线服务。
- 对阶段 3 训练战斗域的大规模重写。

## 2. 架构原则

采用“分层 AI 框架＋可插拔策略”。固定 tick 的单向数据流为：

```text
EncounterController
  -> PerceptionSystem
  -> DecisionPolicy
  -> TacticalPlanner
  -> ActionExecutor
  -> Combat Resolution
  -> SnapshotSystem
```

每个 tick 按以下顺序推进：

1. `EncounterController` 更新遭遇生命周期、有效敌人和区域状态。
2. `PerceptionSystem` 为每个存活敌人生成不可变 `PerceptionSnapshot`。
3. `DecisionPolicy` 根据感知、当前状态和稳定配置选择高层 `EnemyIntent`。
4. `TacticalPlanner` 把意图解析为能力、目标和执行位置，生成 `EnemyActionPlan`。
5. `ActionExecutor` 推进移动、转向、前摇、判定帧、恢复、取消或硬直。
6. 命中请求进入战斗结算层，统一处理无敌、伤害、韧性、附着、反应和死亡。
7. 结算事件反馈给下一 tick 的感知与状态，并发布确定性快照。

必须保持以下边界：

- 决策层不得直接扣血或修改玩家和友军状态。
- 战术层不得推进动作时间线或产生重复命中。
- 执行层不得决定高层意图或修改遭遇生命周期。
- 渲染、HUD 和 ArkTS 不得反向修改 AI 状态。
- 单个敌人不得管理其他敌人的创建、销毁或遭遇完成状态。
- 核心逻辑不得由未校验的字符串分支驱动。

## 3. 核心组件

### 3.1 `EnemyAgent`

`EnemyAgent` 保存单个敌人的稳定实体状态：实体 ID、原型、位置、朝向、出生点、生命、韧性、
护盾、当前 AI 状态、当前意图和执行状态。它组合感知、策略和执行器的结果，但不承担全局
查询、友军遍历或遭遇生命周期。

死亡敌人立即退出感知、决策、碰撞和软锁定候选；最终状态保留在遭遇快照中，直至复位或
停止。

### 3.2 `PerceptionSystem`

感知层只收集事实，不提前做策略判断。`PerceptionSnapshot` 至少包含：

- 当前 tick、敌人和玩家位置、距离、相对角度与朝向差。
- 玩家是否在视野、最后可见 tick、威胁值和是否可达。
- 敌人与出生点、战斗区域边界和安全返回点的关系。
- 最近受击、韧性、硬直和当前动作状态。
- 稳定排序的友军摘要，包括实体 ID、原型、生命、护盾、位置和区域状态。

不提供 `canAttack`、`shouldSupport` 等策略性布尔字段。未来增加听觉、潜行、遮挡或环境感知时，
通过新增事实或版本化扩展实现，不改变决策和执行的职责边界。

### 3.3 `DecisionPolicy`

决策层返回高层 `EnemyIntent`，不返回具体技能。阶段 4 的意图集合为：

- `Idle`
- `Investigate`
- `Chase`
- `Attack`
- `Support`
- `Retreat`
- `ReturnToArea`
- `RecoverFromStuck`

`DecisionPolicy` 是可替换接口。阶段 4 使用数据化效用评分和小型状态约束；后续首领可接入
分阶段策略，未来若出现明确需求，也可以新增行为树或其他策略实现而不改感知与执行层。

相同配置、输入快照和 tick 必须产生相同意图。不得读取墙钟时间或依赖容器迭代顺序。

### 3.4 `TacticalPlanner`

战术层是确定性、无状态的意图解析器：

```text
EnemyIntent + PerceptionSnapshot + AbilityState
  -> EnemyActionPlan
```

它负责：

- 根据意图筛选能力标签。
- 校验距离、角度、冷却、目标规则、友军条件和区域约束。
- 选择具体 `EnemyAbility`、目标实体和期望执行位置。
- 没有合法能力时降级为接近、保持距离、返回区域或待机计划。

能力分数相同时按稳定 ability ID 选择。阶段 4 不实现编队、掩体、复杂路径搜索、GOAP 或行为树
运行时。

### 3.5 `EnemyAbility`

能力使用稳定 ID 和强类型数据定义：

- `id`
- `tags`
- `range`
- `cooldown`
- `windup`
- `active`
- `recovery`
- `weight`
- `cancelPolicy`
- `targetPolicy`

命中形状、位移方式和效果类型使用受控枚举与对应执行器，不使用任意脚本字符串。非法范围、
负时长、倒置窗口、未知标签组合或不兼容目标规则会使整个能力配置被拒绝，不以部分默认值
继续运行。

### 3.6 `ActionExecutor`

执行层统一推进：

- 移动与简单转向。
- 前摇、有效帧、单次命中和恢复期。
- 动作取消、受击硬直、读条中断和破韧中止。
- 返回区域与脱困移动。

执行层只产生规范化移动结果、`HitRequest` 和表现事件。它不得直接修改生命、韧性、护盾或
源附着。大步 tick 必须保留固定命中时刻，且每个动作事务最多产生一次命中。

### 3.7 战斗结算接口

阶段 4 提取敌我共用的命中请求和目标接口，使敌人攻击、玩家攻击、未来陷阱和召唤物可以进入
同一结算边界。战斗结算只认识实体、命中、伤害、韧性、附着和结果，不认识具体 AI 策略。

本阶段不一次性重写阶段 3 的 `CombatController` 为通用 ECS。训练模式通过适配器保留现有行为，
敌人模式逐步复用已验证的伤害、源反应和事件语义。

## 4. 敌人原型

### 4.1 裂爪兽

- 感知玩家后从警戒进入追击。
- 中距离选择冲刺接近，近距离选择明显前摇的爪击。
- 玩家离开战斗区域后停止追击并返回出生区域。
- 与友军距离过近时应用稳定、确定性的分离移动，避免重叠卡死。
- 受击或破韧时进入硬直；死亡后停止所有 AI 与命中生成。

裂爪兽用于验证警戒、追击、攻击前摇、闪避训练、区域返回和基础分离。

### 4.2 辉印祭司

- 优先与玩家保持安全距离，不主动贴身。
- 合法友军缺少护盾且处于支援范围时产生 `Support` 意图，并由战术层选择护盾能力。
- 没有合法支援目标时选择远程攻击或重新定位。
- “审判光束”具有黄色读条、明确前摇和可打断标签。
- 读条期间受到达到阈值的韧性伤害后中断，并进入短暂破绽。
- 不得给自己重复刷新护盾，不得选择死亡或区域外友军。

辉印祭司用于验证友军感知、目标优先级、支援、远程保持距离和打断。

### 4.3 蚀甲守卫

- 高生命和高韧性，移动较慢。
- 正面命中获得减伤，侧后方命中正常结算。
- 近距离选择盾击或重击，中距离缓慢追击。
- 破韧后取消当前动作、移除正面减伤并进入较长硬直。
- 无法到达目标或受边界阻挡时先重新定位；连续失败后返回区域安全点。

蚀甲守卫用于验证朝向伤害、破韧、绕后、崩解和脱困。本阶段不增加阶段切换或首领机制。

## 5. 遭遇与生命周期

统一 `Encounter` 生命周期接口：

```text
start(config, tick)
update(tick, playerState)
reset(tick)
stop()
snapshot()
```

阶段 4 提供：

- `TrainingEncounter`：适配现有 `TrainingTarget + TrainingPulse`。
- `BeastEncounter`：单只裂爪兽。
- `MixedEncounter`：裂爪兽与辉印祭司。
- `GuardEncounter`：单只蚀甲守卫。

`EncounterController` 仅管理遭遇类型、敌人创建与销毁、稳定实体 ID、生命周期、胜利状态、
战斗区域和出生点，并将有效敌人交给软锁定系统。它不管理仇恨、追击、技能选择或伤害。

生命周期规则：

- `start()` 只从干净状态创建完整遭遇；非法配置时拒绝启动且不保留半初始化敌人。
- `reset()` 清除动作、读条、冷却、护盾、硬直、感知记忆和脱困计数。
- `stop()` 后不得继续生成 AI 命中、游戏事件或表现事件。
- 遭遇完成后保留最终快照，不自动切换下一场。
- 训练模式与敌人模式的状态、复位和测试相互隔离。

## 6. 区域约束与脱困

阶段 4 不使用 NavMesh。移动基于平面方向、圆形或凸形战斗区域、简单边界投影与敌人间分离。

- 敌人不得越过战斗区域攻击。
- 玩家离开追击容差后，敌人进入 `ReturnToArea`。
- 分离方向按稳定实体 ID 处理完全重叠，避免零向量和非确定性抖动。
- 连续若干决策周期未显著接近目标时标记不可达，先尝试区域内重新定位。
- 重定位仍失败时进入 `RecoverFromStuck`，移动到配置的安全点并重新警戒。
- 脱困有冷却和次数窗口，禁止每 tick 来回切换。

所有区域投影、不可达阈值、重定位时间和安全点都必须由合法强类型配置提供。

## 7. 快照、桥接与灰盒表现

Native 内部快照拆分为：

- `EncounterSnapshot`：模式、状态、存活数量、完成状态和稳定错误枚举。
- `EnemySnapshot`：实体 ID、原型、位置、朝向、生命、韧性、AI 状态、意图、动作阶段、读条、
  护盾、破韧和脱困状态。
- `DebugSnapshot`：最近决策原因、选择能力和区域约束原因。

对 ArkTS 暂时由现有 `GameSnapshot` 聚合：

- 当前软锁定敌人的完整诊断字段。
- 当前遭遇类型、存活数量和完成状态。
- 最多三个敌人的灰盒渲染数据。
- 最近打断、返回区域和脱困结果。

N-API 不暴露可变容器，使用固定容量数组和有效数量。灰盒表现用形状、颜色、读条与诊断文字
区分三类敌人；关键状态不能只依赖颜色，必须同时有形状、图标或文本差异。

## 8. 错误处理与配置安全

- 未知遭遇、原型、能力或策略 ID 返回稳定错误枚举。
- 非有限数值、负时长、倒置窗口、非正范围和区域外出生点拒绝启动相关遭遇。
- 决策没有合法计划时安全降级为待机、重新定位或返回区域，不执行未校验能力。
- 目标在决策后、执行前死亡时取消计划，不把命中转移给其他实体。
- 软锁定目标失效后按稳定排序重新选择，不保留悬空实体 ID。
- 单个敌人配置错误不得创建部分敌人集合；遭遇启动保持原子性。

所有错误进入 Native 日志和诊断快照。生产逻辑不得静默吞掉未知状态，也不得使用随意默认值
掩盖无效内容配置。

## 9. 测试策略

### 9.1 纯 C++ 单元测试

- 感知：距离、角度、最后可见时间、区域内外、友军状态和不可达目标。
- 决策：状态转换、意图优先级、稳定评分和并列 ID 选择。
- 战术层：意图解析、能力筛选、目标选择、执行位置与安全降级。
- 执行：移动、转向、前摇、单次判定帧、恢复、取消、打断和硬直边界。
- 裂爪兽：警戒、追击、攻击、分离和返回区域。
- 祭司：护盾目标、保持距离、光束读条、打断和破绽。
- 守卫：正面减伤、侧后伤害、破韧中止和脱困。
- 遭遇：启动、完成、停止、复位、稳定实体 ID 与非法配置原子拒绝。

### 9.2 确定性与集成测试

- 相同配置、输入序列和 tick 产生相同意图、计划、事件顺序与快照。
- 不依赖墙钟时间、容器迭代顺序或未播种随机数。
- 大步 tick 不重复命中、不跳过打断窗口。
- 玩家攻击、三源附着、反应、破韧和敌人攻击进入统一结算边界。
- 多目标软锁定稳定，死亡目标立即移除。
- 训练遭遇全部阶段 3 回归测试继续通过。
- 停止重启无动作、读条、命中、护盾、冷却、硬直或感知记忆残留。

### 9.3 构建与真机验收

- 全部既有和新增 C++ 测试、Node 桥接契约通过。
- OHOS arm64-v8a Native 链接和 signed HAP 构建通过。
- 裂爪兽能警戒、追击、前摇攻击、受击、破韧和返回区域。
- 混合战中祭司选择友军护盾，黄色读条可被打断并进入破绽。
- 守卫正面减伤明显，绕后或崩解能够有效破韧。
- 三种敌人遭遇可独立启动和复位，训练模式不回归。
- 移动、相机与战斗按钮保持并发输入，无粘连。

## 10. 实施顺序与提交边界

阶段 4 按依赖顺序拆分：

1. 强类型 AI 配置、感知事实、意图、能力和计划契约。
2. 感知系统、数据化决策策略和战术规划器。
3. 通用动作执行、命中事务、打断与硬直。
4. 敌人实体、区域约束、分离与脱困。
5. 裂爪兽、辉印祭司和蚀甲守卫策略与能力。
6. 遭遇控制、训练适配、多目标软锁定和战斗结算集成。
7. 快照、N-API、灰盒渲染、诊断 HUD 与真机验收。

每一步先增加失败测试，再实现最小通过代码。提交只包含当前任务相关文件；用户现有的
`build-profile.json5` 修改继续排除在阶段 4 提交之外。
