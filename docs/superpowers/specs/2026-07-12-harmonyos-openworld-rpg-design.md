# 《艾瑟兰(Ethelan)》—— 鸿蒙原生开放世界 RPG 综合设计文档

- **文档类型**:综合设计文档(策划 + 技术,两者映射)
- **产出形态**:游戏设计 + 鸿蒙原生技术方案(全量技术规格,接近可立项)
- **技术路径**:方案 1(ArkTS 原生 + 轻量自研引擎)为主干 + 方案 3(渐进式原生渲染)落地节奏 = **原生优先、渐进迁移**
- **鸿蒙特性主线**:性能与续航(方舟编译器 AOT / 鸿蒙图形栈),分布式与元服务为辅
- **日期**:2026-07-12

## 1. 产品定位与世界观

### 1.1 产品定位

鸿蒙原生开放世界动作 RPG(单人/多人在线)。

- **主干(原神式)**:开放世界探索 — 解谜 — 元素战斗 — 角色收集,作为核心体验循环。
- **深度系统(魔兽式)**:天赋树、装备词条/稀有度、地下城副本、世界 Boss、阵营/公会,作为长线成长与社交系统。
- **差异化**:鸿蒙原生实现,主打图形性能与续航(方舟 AOT + 鸿蒙图形栈),辅以分布式协同与元服务即点即玩。

### 1.2 世界观(原创融合,规避 IP 风险)

- **大陆名**:艾瑟兰(Ethelan),由七种元素本源维系。
- **表层(原神味)**:多国度开放探索、元素神明体系、唯美大世界、昼夜与气候。
- **暗线(魔兽味)**:上古「腐蚀军团」自世界裂隙苏醒,带来阵营对立、种族存亡、团队副本 RAID 叙事。
- **设定自洽**:
  - 元素反应 = 战斗机制的世界观根基(七元素本源)。
  - 天赋 / 装备 / 副本 = 各国度为对抗腐蚀军团建立的"备战体系"。
  - 世界裂隙 / 腐蚀节点 = 开放世界高压内容与副本入口的合理来源。

### 1.3 七元素体系

| 元素 | 定位 | 主要反应参与 |
|------|------|-------------|
| 火 Pyro | 高爆发 | 蒸发、融化、超载、燃烧 |
| 水 Hydro | 附着/增伤 | 蒸发、感电、冻结 |
| 风 Anemo | 扩散/聚怪 | 扩散(传播其它元素) |
| 雷 Electro | 群体/持续 | 超载、感电、超导 |
| 草 Dendro | 持续/催化 | 激化、绽放、燃烧 |
| 冰 Cryo | 控制 | 融化、冻结、超导 |
| 岩 Geo | 护盾/结晶 | 结晶(生成护盾) |

## 2. 核心玩法系统设计

### 2.1 探索循环(原神主干)

- **开放世界大地图**:多国度无缝大区,地形/气候/昼夜动态变化;区块流式加载。
- **元素战斗**:七元素基础附着 + 元素反应(蒸发/超载/感电/冻结/扩散/融化/激化等),由 ECS 状态机驱动。
- **角色收集与切换**:4 人小队实时切换;角色定位分为主 C / 副 C / 辅助 / 治疗。
- **探索内容**:解谜机关、"元素之瞳"式收集、开放世界宝箱、世界任务、地区声望。

### 2.2 成长循环(魔兽深度系统)

- **天赋树**:每角色三系天赋(经典树状结构),影响技能形态与数值,支持重置(消耗资源)。
- **装备系统**:
  - 武器 + 圣遗物式套装。
  - 魔兽式词条 / 稀有度:普通 → 精良 → 稀有 → 史诗 → 传说。
  - 随机词条(roll)+ 套装共鸣(2/4 件套效果)。
- **地下城副本**:
  - 5 人小队副本(固定机制 Boss,可重复刷取)。
  - 10 / 20 人 RAID(多阶段机制、掉落分配、周 CD)。
- **世界 Boss / 腐蚀裂隙**:开放世界周期性刷新,伤害归属统计,掉落高价值装备。
- **阵营与公会**:对抗腐蚀军团的阵营声望体系 + 公会(基于鸿蒙分布式好友近场协同组队)。

### 2.3 双循环串联

```
探索循环 ──获取角色/基础资源──▶ 成长循环
   ▲                              │
   │                        获取装备/天赋材料
   │                              │
   └──强化后挑战更高难度腐蚀内容◀──┘
```

- 探索获取角色/资源 → 副本获取装备/天赋材料 → 强化后挑战更高难度腐蚀内容,形成成长飞轮。
- **鸿蒙元服务**:承载"每日副本提醒 / 世界 Boss 刷新"卡片,锁屏/桌面即点即玩。
- **分布式软总线(DSoftBus)**:支撑近场低延迟组队与多端协同。

## 3. 技术架构总览

### 3.1 分层架构(原生优先、渐进迁移)

```
┌─ 表现层 (ArkUI 声明式) ──────────────┐
│   系统 UI / HUD / 元服务卡片 / 设置     │
├─ 桥接层 (NAPI) ───────────────────────┤
│   ArkTS ↔ Native(C++) 调用与内存共享    │
├─ 游戏核心层 (NDK C++ 自研/轻量引擎) ───┤
│   渲染 / 物理 / 战斗状态机 / ECS        │
├─ 平台能力层 (HarmonyOS) ──────────────┤
│   图形栈(ARK) / 分布式软总线 / 元服务   │
│   方舟编译器AOT / 电源&GPU调度 / 音频   │
└─ 服务层 ──────────────────────────────┘
    账号 / 匹配 / 副本服务器 / 热更新CDN
```

### 3.2 关键技术选型

| 维度 | 选型 | 说明 |
|------|------|------|
| UI/逻辑框架 | ArkTS + ArkUI | 声明式 UI、路由、元服务、多端适配 |
| 游戏核心热路径 | Cangjie / C++ (NDK) | 渲染、物理、战斗、ECS,方舟 AOT 编译 |
| 桥接 | NAPI | ArkTS ↔ Native 调用与共享内存 |
| 渲染 | XComponent + OpenGL ES / Vulkan | 自研轻量管线(前向+ / 延迟混合),LOD + 流式加载 |
| 数据驱动 | JSON / 二进制 + Schema 校验 | 天赋/装备/怪物配置外置,热更不改代码 |
| 网络 | QUIC / WebSocket + DSoftBus | 公网权威同步 + 近场分布式协同 |

### 3.3 渐进迁移节奏(方案 3 落地)

- **Phase 0(纯 ArkTS)**:用 ArkUI Canvas 跑通 demo 验证玩法。
- **Phase 1(NAPI 接入)**:渲染/战斗迁到 NDK C++,UI 保留 ArkTS。
- **Phase 2(全面原生)**:ECS 化、自研管线、方舟 AOT 全量编译,达成性能/续航目标。

## 4. 关键难点深挖

### 4.1 开放世界渲染与性能(鸿蒙图形栈 + 方舟 AOT)

- **渲染管线**:XComponent 承载 Native 渲染表面,采用「地形/远景延迟渲染 + 角色/特效前向渲染」混合管线,降低 overdraw。
- **LOD 与流式加载**:基于距离的网格/材质 LOD;HarmonyOS 文件系统 + 协程流式加载区块,避免主线程卡顿。
- **GPU/功耗调度**:利用鸿蒙 GPU 调度与电源档位,按场景动态降分辨率/帧率(战斗 60fps、探索 30fps 自适应),控制发热。
- **方舟 AOT**:游戏核心 C++ 经方舟编译器 AOT 编译,消除解释开销;热点(碰撞检测/状态机结算)内联优化。

### 4.2 元素反应战斗状态机(参考伪代码)

```cpp
enum class Element { Pyro, Hydro, Anemo, Electro, Dendro, Cryo, Geo };

struct ElementStatus {
    Element elem;
    float   duration;   // 附着剩余时间
    int     stacks;     // 附着量(元素量)
};

// 反应表查询:输入两种元素,返回反应结果
Reaction resolve(const ElementStatus& a, const ElementStatus& b) {
    auto key = orderedPair(a.elem, b.elem);
    return REACTION_TABLE.at(key); // 蒸发/超载/感电/冻结/扩散/融化/激化...
}

// ECS 系统:每帧推进附着衰减,命中时触发反应回调
class ReactionSystem : public System {
    void update(float dt) override {
        for (auto& e : query<ElementStatusComponent, HealthComponent>()) {
            e.status.decay(dt);
            if (e.pendingHit) {
                Reaction r = resolve(e.status, e.pendingHit.elem);
                applyReaction(e, r); // 伤害 / 控制 / 扩散
            }
        }
    }
};
```

- 用 ECS 组件 `ElementStatusComponent` 挂在实体上,`ReactionSystem` 每帧结算,**解耦表现与逻辑**,便于服务端重算校验。

### 4.3 魔兽式天赋/装备数据驱动架构

**天赋树**

```cpp
struct TalentNode {
    int id;
    int tier;                 // 层级(解锁前置)
    std::vector<int> requires;// 前置节点
    std::vector<Modifier> effect; // 属性/技能修正

struct TalentTree { int characterId; std::vector<TalentNode> nodes; };
```

**装备与词条**

```cpp
struct Affix   { StatType stat; Range rollRange; float weight; };
struct Equipment {
    BaseStat base;
    std::vector<Affix> affixes; // 随机 roll
    Rarity rarity;              // 普通→精良→稀有→史诗→传说
    int setId;                  // 套装 id
};
struct SetBonus { int setId; int pieces; std::vector<Modifier> effect; }; // 2/4 件套
```

**属性计算管线(分层乘区模型)**

```
最终值 = Base
       × (1 + Σ装备加成)
       × (1 + Σ天赋加成)
       × (1 + Σbuff加成)
       × 元素反应乘区
```

- 全部配置外置(JSON / 二进制),热更仅下发配置,不改代码;运行时用 `Modifier` 聚合器计算最终属性。

## 5. 网络同步、资源热更、性能实测指标

### 5.1 网络同步方案

- **副本(5/10/20人)**:服务器权威 + 客户端状态同步。Boss 机制关键帧由服务器下推,客户端做插值/预测补偿延迟;QUIC 传输降低丢包重传影响。
- **近场组队**:鸿蒙分布式软总线(DSoftBus)做同账号/同局域网设备低延迟协同,手机与平板可互为输入输出。
- **开放世界**:AOI(兴趣区域)分线,玩家只同步视野内实体;世界 Boss 用广播 + 伤害归属统计。
- **一致性/反作弊**:关键战斗逻辑(元素反应/伤害)服务端重算校验。

### 5.2 资源包与热更新

- **分包架构**:基础包(必装)+ 资源分包(国度/副本按需下载),基于鸿蒙 `Hap` 模块化 + 资源管理器。
- **热更**:
  - 配置/数值走 CDN 差分下发(bsdiff),无需发版。
  - 资源走鸿蒙后台 `ability` 静默下载,元服务卡片提示更新进度。
- **版本兼容**:配置带 schema 版本号,旧客户端安全忽略未知字段。

### 5.3 性能与续航实测指标(目标值)

| 指标 | 探索场景 | 20 人 RAID |
|------|---------|-----------|
| 帧率 | 30–60 自适应 | ≥ 30 |
| 功耗 | ≤ 4.5W | ≤ 6W |
| 机身温度 | ≤ 40°C | ≤ 43°C |
| 内存占用 | ≤ 1.8GB | ≤ 2.4GB |
| 首屏冷启 | ≤ 3s(基础包) | — |
| 包体 | 基础包 ≤ 800MB | 全资源 ≤ 12GB |

### 5.4 功耗/发热控制策略

- 动态分辨率 + 帧率档位切换。
- 后台/低负载场景降频。
- 方舟 AOT 减少 CPU 峰值。
- GPU 长帧合并,减少无效绘制。

## 7. 风险与鸿蒙原生价值总结

### 7.1 主要风险与对策

| 风险 | 对策 |
|------|------|
| 图形自研成本高 | 渐进迁移(Phase 0/1/2),先验证玩法再加重投入;复用开源轻量渲染组件 |
| 方舟 AOT 对 C++ 游戏核心的收益边界 | 仅热路径走 NDK,UI/逻辑框架留 ArkTS,平衡开发效率与性能 |
| 分布式组队网络复杂度 | 近场用 DSoftBus、公网用成熟 QUIC,服务端权威兜底 |
| 包体与功耗 | 分包 + 动态资源 + 帧率/分辨率自适应,设 §5.3 硬指标卡点 |
| 合规与商店审核 | 元服务/分布式能力按鸿蒙规范接入,预留多设备适配 |

### 7.2 鸿蒙原生价值总结(为什么是鸿蒙原生而非移植)

- **性能原生**:方舟 AOT + 鸿蒙图形栈 + GPU/电源调度,直接支撑开放世界渲染与续航目标。
- **分布式协同**:DSoftBus 实现手机/平板/车机/智慧屏无缝流转与近场组队,友商难以复制的体验。
- **元服务即点即玩**:副本/日常/世界 Boss 以元服务卡片触达,降低启动摩擦、提升留存。
- **一次开发多端部署**:ArkUI 多端适配,降低多形态设备覆盖成本。
- **生态位**:作为"鸿蒙原生标杆级开放世界 RPG",抢占原生游戏生态心智。
