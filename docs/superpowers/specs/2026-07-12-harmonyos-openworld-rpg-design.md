# 《艾瑟兰（Ethelan）》——鸿蒙原生开放世界动作 RPG 综合设计文档

- **文档类型**：MVP 垂直切片 + 鸿蒙原生技术方案 + 长期方向说明
- **当前阶段**：立项前验证
- **技术主线**：ArkUI + XComponent + HarmonyOS NDK C++，原生渲染骨架先行
- **产品原则**：先验证原创玩法与鸿蒙原生性能，再决定联网和内容规模扩张
- **日期**：2026-07-12

---

## 1. 产品定位与范围

### 1.1 产品定位

《艾瑟兰》是一款面向 HarmonyOS 的原创开放世界动作 RPG。游戏以自由探索、即时战斗、环境解谜和角色成长为核心体验，并以原创的“三源共鸣”构建战斗与世界规则。

参考产品仅用于分析高层体验：开放世界的探索节奏、动作战斗的反馈，以及大型角色扮演游戏的长期成长和团队挑战。名称、视觉轮廓、成长资源、战斗规则、界面布局和叙事组织均须独立设计，不复刻任何既有作品的具体表达。

### 1.2 两层产品结构

#### MVP 垂直切片（当前承诺）

MVP 是一个可连续体验 15～30 分钟的单机探索战斗样板，用于验证玩法原创性、技术可行性和目标设备表现。

- 1 个可探索的小型区域，包含地形高差、环境机关和一条引导路线。
- 1 个可控角色，具备移动、冲刺、闪避、普通攻击、两个主动能力和一个终结能力。
- 3 类敌人：近战压迫型、远程牵制型、源质干扰型。
- 1 个多阶段首领，要求玩家综合使用三源共鸣、闪避和削韧机制。
- 1 条完整闭环：探索 → 遭遇 → 战斗 → 获得构件 → 强化能力 → 挑战首领。
- 1 组基础内容：源痕收集物、环境机关、遗物构件和营地升级点。

MVP 明确不包含角色抽取或付费、公会、阵营、玩家匹配、多人副本、世界 Boss、完整 MMO 后端及跨设备联机。

#### 长期产品愿景（非 MVP 承诺）

当 MVP 的留存意向、战斗反馈、内容生产效率和真机性能达到评审标准后，再逐项评估以下能力：

- 扩展为多区域开放世界，增加气候、昼夜和区域生态。
- 增加可控角色、专精路线、遗物构筑和合作战斗。
- 建设 4 人合作副本、世界事件、世界 Boss、公会及大型团队挑战。
- 建设权威服务器、匹配、反作弊、运营配置和在线内容分发体系。
- 增加跨设备任务接续、辅助屏幕或控制器协同等鸿蒙特色体验。

长期能力必须单独立项，不得作为 MVP 验收的隐含条件。

## 2. 世界观与原创性约束

### 2.1 世界设定

艾瑟兰大陆由遍布地下的“世界脉网”维持。古代文明曾利用脉网塑造城市和生态，但一次称为“静默断层”的灾变使部分脉网失控，形成不断侵蚀环境的荒化地带。

玩家扮演能够观测并调谐源质的“巡脉者”，进入断层区域修复节点、回收失落构件，并调查灾变背后的历史。主要冲突来自不同势力对脉网用途的分歧，而非复刻既有作品的神明国度、阵营战争或军团入侵结构。

### 2.2 三源共鸣

| 源质 | 战斗功能 | 环境功能 |
|------|----------|----------|
| 辉印 | 标记目标、放大弱点、提高后续攻击收益 | 显示隐藏路径和机关纹路 |
| 脉流 | 牵引、传导、位移和区域控制 | 驱动装置、改变流体或风道 |
| 蚀质 | 削韧、持续侵蚀、改变目标状态 | 清除或重塑受侵蚀障碍 |

两种源质按命中顺序组合为共鸣效果：

- **折光（辉印 + 脉流）**：展开短时弱点窗口，并将部分伤害传导至邻近目标。
- **凝滞（脉流 + 蚀质）**：生成减速区域，延长目标失衡时间。
- **崩解（蚀质 + 辉印）**：快速削减韧性，击破后暴露核心。
- **共鸣爆发**：在限定时间内依次完成三种源质命中后触发；清除已有附着并产生一次与敌人状态相关的终结效果。

组合顺序、附着量和目标抗性影响效果强度；共鸣不是简单的固定倍率表。

### 2.3 成长与收集

- **源痕**：分布在区域中的探索收集物，用于解锁世界信息和营地能力。
- **遗物构件**：改变技能行为或战斗节奏的装备模块，不采用固定的既有作品套装结构。
- **巡脉专精**：围绕机动、共鸣和生存形成三条轻量成长路径；MVP 仅提供少量互斥选择，用于验证构筑差异。
- **营地升级**：使用探索和首领奖励解锁恢复、扫描与构件调校能力，形成区域内的成长反馈。

### 2.4 原创性门禁

每个进入制作的角色、敌人、系统和场景均需通过以下检查：

- 名称、轮廓、配色、动作节奏和特效语言不能直接对应参考作品中的具体对象。
- 战斗规则必须能用本作世界观解释，且至少包含一项独有的输入、状态或取舍。
- 收集物、装备槽位、成长货币和掉落界面不得仅通过改名复用成熟产品结构。
- UI 信息层级和布局从本作操作场景出发设计，不照搬参考作品截图。
- 文案使用“体验目标”描述参考方向，避免在正式设计中继续使用其他产品的专有式称呼。

## 3. MVP 核心体验

### 3.1 探索—战斗—成长闭环

```text
探索区域与发现源痕
        ↓
利用三源能力解开机关
        ↓
遭遇敌人并完成共鸣战斗
        ↓
获得遗物构件与营地资源
        ↓
调整构筑并挑战区域首领
```

探索必须为战斗或成长提供有效信息，战斗奖励也必须改变后续探索或首领策略，避免形成互不相关的系统堆叠。

### 3.2 操作与战斗

- 移动采用第三人称自由视角，支持触屏虚拟摇杆和外接控制器映射。
- 闪避提供短暂规避窗口，成功规避强攻击后获得少量源能奖励。
- 普通攻击负责基础输出和源能积累；主动能力负责施加三源状态。
- 敌人具有生命、韧性、源质抗性和攻击意图；首领在阶段切换时改变弱点和场地规则。
- 战斗逻辑使用固定逻辑 tick，渲染帧率与逻辑结算解耦，为回放、调试和未来服务器校验预留条件。

### 3.3 MVP 内容边界

MVP 不以“地图大小”或“系统数量”为完成标准，而以完整、稳定、可重复验证的体验为标准。下列内容只制作支撑闭环所需的最小集合：

- 一个区域主题和一套环境材质语言。
- 一条主路径和两条可选探索支路。
- 三种普通敌人、一种精英变体和一个首领。
- 三件具有明确玩法差异的遗物构件。
- 一次营地升级和一次构筑调整。

## 4. 鸿蒙原生技术架构

### 4.1 分层架构

```text
┌─ ArkUI 表现与系统交互层 ─────────────────────┐
│ 菜单 / HUD / 设置 / 账号接入 / 系统弹窗       │
├─ N-API 边界层 ───────────────────────────────┤
│ 生命周期 / 输入事件 / 状态快照 / 批量命令     │
├─ HarmonyOS NDK C++ 游戏核心 ─────────────────┤
│ 帧循环 / 相机 / 渲染 / 资源 / 碰撞 / 战斗 ECS │
├─ HarmonyOS 平台能力层 ───────────────────────┤
│ XComponent / NativeWindow / 图形 API / 音频   │
│ Game Service Kit / Graphics Accelerate Kit     │
└─ 可选在线与内容服务层（MVP 后评估）───────────┘
  账号后端 / 匹配 / 权威战斗 / CDN / 运营配置
```

### 4.2 技术选型

| 维度 | MVP 选型 | 责任边界 |
|------|----------|----------|
| 系统 UI | ArkTS + ArkUI | 页面、HUD、设置、系统能力和无障碍信息 |
| 渲染承载 | XComponent + NativeWindow | 创建并管理 Native 渲染表面 |
| 游戏核心 | HarmonyOS NDK C++ | 帧循环、相机、碰撞、资源和战斗 ECS |
| 桥接 | N-API | 生命周期和粗粒度数据交换，避免逐实体跨语言调用 |
| 图形 API | 通过 M0 技术探针确定 | 在目标设备验证能力、驱动稳定性和工具支持后锁定 |
| 配置 | 版本化 Schema + 开发期 JSON + 发布期二进制 | 保证配置校验、兼容和加载效率 |

ArkTS 代码由 ArkCompiler 编译和优化；C++ 游戏核心使用 HarmonyOS NDK 原生工具链构建，并通过 LTO、PGO、数据布局、内存池、SIMD 可行性验证和热点分析优化。二者不得统一表述为“C++ 经方舟编译器 AOT”。

### 4.3 技术演进

- **M0 技术探针**：直接建立 ArkUI、XComponent、NativeWindow 和 NDK C++ 最小骨架，验证输入、相机、资源加载、渲染循环、暂停恢复及设备兼容性。
- **M1 战斗样板**：在 Native 游戏核心中实现固定 tick、碰撞、三源共鸣、敌人状态和调试记录。
- **M2 垂直切片**：加入区域流式加载、完整玩法闭环、内容配置和存档。
- **M3 鸿蒙验证**：接入所需游戏服务能力，完成三档设备的性能、功耗和稳定性测试。
- **M4 长期能力评审**：依据 MVP 数据决定是否投资公网联机、多人副本、角色扩展和大世界生产。

不再使用 ArkUI Canvas 制作随后整体废弃的游戏原型。低成本玩法验证可使用灰盒资源，但必须运行在最终的 Native 帧循环和输入边界上。

## 5. 三源共鸣的确定性战斗模型

### 5.1 概念数据结构

```cpp
enum class SourceType { Radiance, Current, Corruption };

struct SourceAura {
    SourceType type;
    FixedPoint amount;
    Tick expireAt;
    EntityId applier;
};

class SourceAuraContainer {
public:
    void apply(const SourceAura& aura);
    void decay(Tick now);
    Span<const SourceAura> activeAuras() const;
};

struct HitEvent {
    EntityId attacker;
    EntityId target;
    AbilityId ability;
    SourceType source;
    FixedPoint sourceAmount;
    FixedPoint baseDamage;
    Tick tick;
    uint32_t sequence;
};

struct CombatResult {
    FixedPoint damage;
    FixedPoint poiseDamage;
    Optional<ResonanceType> resonance;  // ResonanceType ∈ {折光, 凝滞, 崩解, 共鸣爆发}
    Vector<GameplayEvent> gameplayEvents;
    Vector<PresentationEvent> presentationEvents;
};
```

### 5.2 结算规则

1. 按 `tick`、事件优先级和 `sequence` 对命中排序。
2. 校验目标状态、能力合法性和重复事件。
3. 应用或刷新源质附着量，并按配置化优先级寻找可触发组合。
4. 消耗对应附着量，生成伤害、削韧、控制和场地事件。
5. 战斗状态只消费 `GameplayEvent`；渲染、音效和震动只消费 `PresentationEvent`。
6. 将关键输入和结果写入调试记录，支持确定性回放和差异定位。

`SourceAuraContainer` 支持多源共存、独立衰减、施加者追踪和顺序敏感结算。MVP 不建设服务器，但命中事件和结果必须可序列化，为未来权威服务器重算保留接口。

## 6. 内容、存档与资源交付

### 6.1 数据驱动

- 角色能力、敌人、共鸣、构件和掉落均使用带版本号的 Schema。
- 开发构建读取便于审查的 JSON；发布构建转换为经过校验的二进制数据。
- 配置加载失败时拒绝进入相关内容，并输出明确的版本、资源 ID 和字段错误，不使用不可追踪的默认数值继续运行。
- 影响战斗确定性的配置必须参与版本指纹计算并写入存档和调试记录。

### 6.2 存档

MVP 使用本地版本化存档，记录区域进度、营地等级、构件和设置。写入采用临时文件、校验和及原子替换，保留最近一次有效备份。云存档和跨端接续在 M4 单独评估。

### 6.3 应用包与游戏资源

- 应用安装包包含启动所需代码、基础 UI、首屏资源和最低可运行内容。
- 大型游戏资源存放在独立资源目录，通过资源清单描述版本、大小、哈希和依赖关系。
- 后续内容分发采用 CDN、分块下载、断点续传、完整性校验、磁盘空间预检和失败回滚。
- 代码与原生库更新遵循应用发布渠道，不通过资源热更替换可执行逻辑。
- HAP、HAR、HSP 用于应用模块和共享代码组织，不作为大型地图资源的抽象。
- 后台下载必须遵循 HarmonyOS 生命周期、网络、功耗和用户授权约束，不承诺无条件静默执行。

MVP 只实现本地打包资源的加载和校验；CDN 下载在长期内容服务立项后实施。

## 7. 在线能力与鸿蒙特色能力边界

### 7.1 在线能力

MVP 不建设公网联机。未来多人玩法统一采用权威服务器：客户端提交输入和意图，服务器负责关键状态、伤害、掉落和首领机制结算，并处理匹配、断线重连、AOI 和反作弊。

DSoftBus 不用于公网匹配或多人战斗同步。若未来验证存在明确用户价值，可用于同一用户设备间的任务接续、辅助信息屏、控制器协同或局域设备发现，并必须具备单设备降级路径。

### 7.2 元服务

元服务仅作为主游戏的轻量入口，可候选承载预约、活动提醒、进度摘要和拉起主应用。元服务不承载开放世界主体、不替代游戏安装包，也不作为代码或大型资源热更新通道。

### 7.3 游戏服务与系统能力

根据产品发布要求和目标 SDK 能力，分阶段评估接入：

- Game Service Kit：华为账号快捷登录、实名认证、防沉迷和游戏场景感知。
- Graphics Accelerate Kit：根据正式开放接口验证游戏负载感知和画质自适应能力。
- 控制器、音频、振动及无障碍能力：在 M0/M1 建立兼容性清单和降级策略。

文档只承诺经过目标 SDK 和真机验证的公开能力，不直接承诺应用可以控制 GPU 频率、电源档位或其他系统级调度策略。

## 8. 性能、稳定性与验收

### 8.1 测试设备分档

三档目标设备清单由立项阶段(早于 M0)确定,并作为 M0 技术探针的验收基线。每档记录具体机型、芯片、内存、系统版本、分辨率和刷新率；未确定机型前不填写绝对功耗、温度、内存或包体承诺。

### 8.2 固定测试条件

- 使用锁定的游戏版本、资源版本、画质档和渲染分辨率。
- 记录环境温度、初始电量、网络状态、屏幕亮度和后台进程条件。
- 预热后连续运行 30 分钟，分别覆盖探索、普通战斗和首领战斗。
- 性能报告同时给出原始采样、统计方式、测试设备和异常说明。

### 8.3 核心指标

| 类别 | 指标 |
|------|------|
| 流畅度 | P50/P95 帧时间、1% low、卡顿次数与最长卡顿 |
| 内存 | 稳态占用、峰值、区域切换峰值和退出后的资源回收 |
| 功耗温升 | 30 分钟稳态功耗、温升曲线和降频发生时间 |
| 启动加载 | 冷启动、进入可操作状态、区域切换和首领重试耗时 |
| 稳定性 | 崩溃、黑屏、前后台切换、来电中断和低内存恢复 |
| 资源 | 安装包、首屏资源、区域资源和峰值磁盘占用 |

M0 产出三档设备基线；M1～M3 再基于基线确定量化门槛。未经真机数据支持的数值不得作为硬指标写入立项结论。

### 8.4 MVP 验收标准

- 可以从启动连续完成探索、机关、三类敌人、成长选择和首领挑战。
- 三源共鸣在固定输入回放中产生一致结果，调试记录能够定位结算差异。
- 前后台切换、暂停恢复和资源重载不丢失有效存档、不出现持续黑屏。
- 最低档设备达到 M0 确定的帧时间和稳定性门槛，推荐档设备达到目标画质体验。
- 至少完成一轮玩家测试，验证战斗规则可理解、共鸣选择有差异、探索奖励有意义。

## 9. 工程结构与质量保障

### 9.1 DevEco 工程结构

```text
my-world/
├─ entry/                         # 主 HAP：生命周期、ArkUI、XComponent
│  └─ src/main/
│     ├─ ets/                     # 菜单、HUD、设置、系统能力
│     └─ cpp/                     # N-API 注册和构建入口
├─ native/
│  ├─ engine/                     # 帧循环、渲染、资源、输入、碰撞
│  ├─ gameplay/                   # 三源共鸣、敌人、构件、成长
│  └─ platform/harmony/           # NativeWindow、音频、文件和生命周期适配
├─ config/                        # Schema 与开发期配置
├─ assets/                        # MVP 本地资源及清单
├─ tests/                         # 确定性、配置、存档和玩法测试
└─ automation/                    # Hvigor、检查脚本和性能采集
```

长期在线服务在正式立项后建立独立工程，不在 MVP 仓库中预建空壳服务。

### 9.2 质量门禁

- ArkTS 静态检查、C++ 编译告警和基础代码规范检查。
- 三源反应表、事件顺序、附着消耗和确定性回放单元测试。
- Schema、资源清单、哈希和配置引用完整性检查。
- 存档升级、损坏回退、前后台切换和低磁盘场景测试。
- 三档真机的帧时间、内存、功耗和稳定性回归。
- 原创性评审记录作为角色、敌人、场景和 UI 进入制作的前置材料。

## 10. 里程碑与决策门

| 里程碑 | 主要交付 | 决策门 |
|--------|----------|--------|
| M0 技术探针 | Native 渲染表面、输入、相机、资源加载、持续帧循环 | 技术路径在三档目标设备上可运行且可分析 |
| M1 战斗样板 | 1 个角色、三源共鸣、3 类敌人、1 个灰盒首领 | 战斗可理解、可回放，共鸣带来可观察的策略差异 |
| M2 垂直切片 | 1 个小型区域和探索—战斗—成长闭环 | 15～30 分钟体验完整，内容生产流程可重复 |
| M3 鸿蒙验证 | 游戏服务接入、真机性能、功耗和稳定性报告 | 最低档稳定，推荐档达到目标体验，无阻断性平台风险 |
| M4 长期能力评审 | MVP 数据、成本估算和长期路线候选 | 决定继续单机内容、建设合作副本或扩大开放世界 |

未通过当前决策门时，优先削减内容复杂度或修正核心体验，不提前启动下一阶段的大规模内容生产。

## 11. 主要风险与应对

| 风险 | 应对 |
|------|------|
| 自研渲染与开放世界成本失控 | M0 先验证最小 Native 骨架；M2 只制作一个小型区域，不承诺无缝多国度 |
| 玩法与参考作品过于相似 | 使用三源共鸣和原创世界规则；设置名称、视觉、系统和 UI 原创性门禁 |
| ArkTS 与 C++ 边界产生性能问题 | N-API 使用粗粒度命令和状态快照，避免逐实体、逐帧对象往返 |
| 战斗逻辑未来无法联网 | 固定 tick、可序列化事件、确定性回放和表现分离从 M1 开始建立 |
| 性能目标脱离设备现实 | M0 先建立三档真机基线，再填写量化门槛 |
| 资源更新破坏版本一致性 | 资源清单、哈希、依赖校验、空间预检和失败回滚；代码更新走正式渠道 |
| 鸿蒙能力描述或接入超出公开边界 | 以目标 SDK 正式文档和真机验证为准，为每项能力设计降级路径 |
