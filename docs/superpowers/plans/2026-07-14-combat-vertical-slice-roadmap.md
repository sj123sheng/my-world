# 艾瑟兰战斗垂直切片实施路线

**依据规格：** `docs/superpowers/specs/2026-07-14-combat-vertical-slice-design.md`

垂直切片按依赖顺序拆为六个可独立验收的实施计划。每个阶段必须保持真机已验证的 EGL/GLES3 路径可用，并在进入下一阶段前通过自身自动化测试。

| 阶段 | 独立交付物 | 依赖 | 验收出口 |
|---|---|---|---|
| 1. 基础循环与事件 | 线程安全输入、固定时间步、类型化事件、线程安全快照 | 现有 Native 渲染循环 | macOS C++ 测试通过，现有 GLES3 真机路径不变 |
| 2. 输入与自由相机 | 双区触控、虚拟摇杆、自由第三人称相机、软锁定输入语义 | 阶段 1 | 输入/相机确定性测试和真机双指操作通过 |
| 3. 角色战斗与三源 | 四段普攻、闪避、体力、三源技能、反应和共鸣终结技 | 阶段 2 | 战斗状态与三源完整单元/集成测试通过 |
| 4. 敌人 AI | 感知/决策/执行三层 AI，三类敌人和战斗区域约束 | 阶段 3 | AI 状态转换、打断、破韧和脱困测试通过 |
| 5. 首领与关卡 | 三阶段卡洛恩、遭遇流程、战斗门、补给和快速重试 | 阶段 4 | 首领完整推进、阶段单次触发和重试复位通过 |
| 6. HUD、表现与优化 | 移动 HUD、关键 VFX/音频事件、性能降级和真机报告 | 阶段 5 | 8–12 分钟完整流程、性能与稳定性门槛通过 |

阶段 1 的详细计划见：`docs/superpowers/plans/2026-07-14-combat-foundation.md`。

后续每个阶段开始前，根据上一阶段落地后的真实接口编写对应详细计划，避免在当前代码尚未形成时锁死错误签名。阶段边界和验收出口已经固定，后续详细计划不得扩大规格范围。

## 阶段 2 验收状态（2026-07-15）

- 实现提交：`5c18eb1` 至 `d13b0e1`，覆盖双区多指输入、相机相对移动、自由相机、
  确定性软锁定、固定循环集成、桥接快照以及 OHOS Native C++17 构建修复。
- 最终审查修复：相机 target/yaw/pitch/distance 已进入 GL 与软件灰盒渲染，网格、场景物件、
  粒子和玩家共用世界到视图变换；触控仅由 ArkTS `changedTouches` 经 N-API 生产；输入
  sequence 分配与队列 push 已在线性化临界区内完成；非正相机距离配置整体回退安全默认。
- 最终复审补充：世界到视图 yaw 使用角色控制器 camera right/forward 基的严格逆变换；
  跨模块测试覆盖任意 yaw 的移动方向和软锁定前轴。玩家、粒子使用随 distance 缩放、
  不受 pitch 拉伸的屏幕 billboard，GL/软件路径在非方形视口中保持相同像素圆形。
- 自动化通过：20/20 个 C++ 测试与 1/1 个 Node 桥接契约测试退出码均为 0。
- Native 链接通过：OHOS arm64-v8a `native_game` 完整编译并链接为 aarch64 ELF 共享库。
- HAP 构建通过：Hvigor `assembleHap` 返回 `BUILD SUCCESSFUL`，生成 unsigned HAP；未配置
  签名，因此不包含安装验证。
- 真机待验：双指并发、单指释放隔离、相机稳定性与目标越界清理均未执行。阶段 2 的代码、
  自动化和构建收口已完成，但表格中的“真机双指操作通过”出口仍未完成，进入阶段 3 前需补验。

## 阶段 2 真机输入链路修正（2026-07-15）

- 真机复验确认单指 UI Test 与双指 uinput 均能稳定注入，但带 `libraryname` 的 Native
  XComponent 不触发页面 `.onTouch`，旧 ArkTS 单一生产者方案因此无法驱动画面。
- 生产输入改为 Native `DispatchTouchEvent` 单一来源：逐个处理 `touchPoints`，每点使用
  自身 `type/id/x/y`；`numPoints == 0` 时兼容回退到事件顶层字段。
- ArkTS 页面删除 `.onTouch` 与 `pushInput` 调用；N-API `pushInput` 仅保留为测试或未来外部
  输入入口，不参与当前页面生产链路，避免双重生产。
- 自动化、Native 链接与 signed HAP 构建完成后，仍需在设备上复验 HUD 移动/相机值与画面。
