# 阶段 4 敌人 AI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在阶段 3 固定 tick 战斗域上交付可扩展、可维护、确定性的分层敌人 AI、三类灰盒敌人和三种可独立验收的敌人遭遇。

**Architecture:** EncounterController 按固定 tick 编排 PerceptionSystem → DecisionPolicy → TacticalPlanner → ActionExecutor → Combat Resolution → SnapshotSystem。训练遭遇通过适配器保留 TrainingTarget + TrainingPulse；敌人遭遇复用命中、伤害、韧性、源附着和事件语义。

**Tech Stack:** C++17、HarmonyOS N-API、ArkTS、OpenGL ES 3、Node.js 合约测试、DevEco Hvigor、HDC 真机验证

## Global Constraints

- 不引入 NavMesh、行为树运行时、GOAP、脚本运行时、网络同步或 Mod 加载。
- 不实现掉落、战斗门、波次推进、失败重试或首领阶段；这些属于阶段 5。
- 不制作正式模型、动画、VFX、音频或最终 HUD；这些属于阶段 6。
- 感知、决策、战术、执行、结算和快照保持单向依赖，ArkTS 不参与 AI 决策。
- 相同配置、输入序列和 tick 必须产生相同意图、计划、事件顺序和快照。
- 不读取墙钟时间，不依赖容器迭代顺序，不使用未播种随机数。
- N-API 不暴露可变容器；敌人快照固定容量为 3。
- 训练模式的阶段 3 自动化与真机语义不得回归。
- 不修改、暂存或提交用户现有的 build-profile.json5。
- 每个提交必须包含 Prompt: 进入阶段4并优先保证扩展性与可维护性。

## File Structure

新增 native/gameplay/ai/：

- enemy_ai_types.h：稳定枚举、感知事实、意图、能力、计划和快照值类型。
- enemy_ai_config.h：强类型 AI、能力、原型、区域和遭遇配置验证。
- perception_system.*：生成不可变感知快照。
- decision_policy.*：由事实选择高层意图。
- tactical_planner.*：由意图选择能力、目标和执行位置。
- action_executor.*：移动、转向、前摇、有效帧、恢复、打断与命中事务。
- combat_region.*：区域投影、稳定分离、不可达判定和脱困。
- enemy_agent.*：组合单个敌人的状态、策略和执行器。
- enemy_archetypes.*：三类敌人的合法默认配置。
- encounter_controller.*：遭遇生命周期、稳定实体集合、胜利和训练适配。

现有 Loop、CombatController、SoftTargeting、Surface、GameSnapshot 与 N-API 只承担集成，不把 AI 逻辑堆入桥接或渲染层。
